<h2><a id="user-content-http" class="anchor" aria-hidden="true" href="#http"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>HTTP</h2>
<p><em><strong>That's all HTTP is: an agreement on how to format a string request.</strong></em></p>
<p>HTTP is what's known as an application layer protocol, which means it's just a convention or agreement between two parties. For example, if I ran a historical weather service, we might agree that when you need some data from me, you can request the date and zip code and I can give you the weather for that date in that zip code.</p>
<p>But wait, we need to be more specific, because what if you request "Jan 3rd, 2012 in zip code 34567". Will my service understand that? We need some application protocol, so we'll agree that this should be the request format "MM/DD/YYYY; ZIPCODE".</p>
<p>As long as you send a request in that specific format, I can write a program on my end to parse that request and send you the requested data back.</p>
<p><em>That's all HTTP is: an agreement on how to format a string request.</em></p>

<p><em>HTTP stands for Hypertext Transfer Protocol</em></p>
<ul>
<li>It is a system of rules, a protocol, that serve as a link between applications and the transfer of hypertext documents.</li>
<li>Stated differently, it's an agreement, or message format, of how machines communicate with each other</li>
<li>HTTP follows a model where a <strong>client</strong> makes a <strong>request</strong> to a server and waits for the <strong>server</strong> to send a <strong>response</strong>. Hence, it's referred to as a request response protocol.</li>
</ul>

<ul>
<li>A start-line describing the requests to be implemented, or its status of whether successful or a failure. This start-line is always a single line.</li>

<p>If handled incorrectly, a user could inject malicious code into HTTP requests. Some areas to look out for to mitigate risk include:</p>
<ul>
<li>Avoid using parameters to construct file paths if at all possible.</li>
<li>Use whitelisting to explicitly define what is allowed.</li>
<li>Try to think like a person that is attempting to alter the behavior of your software. Is it possible to introduce an unexpected value into the system through expected means such as parameters?</li>
</ul>
<p><em>Basically, we need to really pay attention to our input and be sure to both sanitize the HTML as well as prepare for how we handle bad input.</em></p>
<p>More on security:</p>
<p>Is using POST as the HTTP method for a request more secure than GET? Why or why not?</p>
<ul>
<li>No. Any request sent as plain text, regardless of the HTTP method used, is equally vulnerable to being seen while in transit on the network.</li>
</ul>
<p>How can a web application be secured so that any data being sent between the client and server can not be viewed by other parties?</p>
<ul>

<h3><a id="user-content-what-is-the-session-where-it-is-stored-how-it-is-used" class="anchor" aria-hidden="true" href="#what-is-the-session-where-it-is-stored-how-it-is-used"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What is the session? Where it is stored? How it is used?</h3>
<p><em>Definition:</em></p>
<p>A session can be defined as a server-side storage of information that is desired to persist throughout the user's interaction with the web site or web application.</p>
<p><em>Explanation</em></p>
<p>Instead of storing large and constantly changing information via cookies in the user's browser, only a unique identifier is stored on the client side (called a "session id"). This session id is passed to the web server every time the browser makes an HTTP request (ie a page link or AJAX request). The web application pairs this session id with it's internal database and retrieves the stored variables for use by the requested page.</p>
<p><em>From: <a href="http://www.lassosoft.com/Tutorial-Understanding-Cookies-and-Sessions" rel="nofollow">http://www.lassosoft.com/Tutorial-Understanding-Cookies-and-Sessions</a></em>
<em>I really like the above definition and explanation</em></p>
<p>I think like a "session" in real life, defined as <em>a period devoted to a particular activity</em>, it implies a timeframe where we are doing something together. In this case, a client and a server are focused on persistent communication while using HTTP requests and responses to communicate.</p>
<p>The session id is generally stored as a cookie in a browser.</p>
<p>It's literally just used to make sure we are working with the same person within the same session so that we can communicate back and forth while giving the appearance of persistence, when in actuality each HTTP request is independent of the prior one. We only have the memory of the prior request because we are tracking the session id and accessing "memory"  from the server.</p>
<p>It's like if a bunch of people in a room are speaking, but we only identify one person in the room to listen to and we can only remember the prior words spoken by that person if we are focusing on that one person and actively listening to them. The words themselves don't remember the prior words, but words chained together from a known source can provide meaning.</p>
